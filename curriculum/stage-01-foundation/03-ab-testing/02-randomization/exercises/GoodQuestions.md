# 好问练习 - 随机化与分流

## Q1: 两个独立实验使用相同哈希种子，73%的实验A实验组用户同时落入实验B实验组——这为什么会发生？ [L1 概念层]

你是某大型平台的实验平台负责人。两个独立实验A（新按钮颜色）和B（新推荐算法）理论上应该正交（互不干扰），但分析发现实验A的实验组用户有73%同时落入实验B实验组。排查后发现两个实验使用了相同的哈希种子。你需要向团队解释：为什么哈希种子相同会导致分流相关？正交分流的数学基础是什么？

## Q2: 用户ID从自增整数改为UUID后，SRM持续告警——是ID的问题还是分流算法的问题？ [L2 混淆层]

某出行平台将用户ID从自增整数改为UUID。实验平台的分流逻辑基于"用户ID末位数字取模"，UUID末位数字分布不均匀，导致新用户被系统性偏向某些实验组，SRM告警持续两周。业务方质疑："是UUID设计有问题，还是我们的分流算法太简陋？"你需要澄清：分流算法应该对ID的"表面随机性"做什么假设？正确的分流算法应该依赖什么？

## Q3: 我们用"时间戳取模"分流，早高峰用户全进实验组——时间分流为什么看似公平实则危险？ [L3 错误层]

你们为一个临时活动实验采用"时间戳取模"分流：用户进入页面时的时间戳秒数 % 2 == 0 为实验组。上线当天早8点，实验组流量占比68%，触发SRM。更隐蔽的是：早高峰用户本身是高价值用户（通勤场景），实验组系统性富集了高价值人群。这个错误为什么不是"偶发波动"，而是"时间分流固有的结构性偏误"？

## Q4: 同一用户用手机和iPad访问，被分到不同组——设备切换问题该怎么解决？ [L4 应用层]

某内容平台的用户级实验（用户ID分流）中，发现同一用户用手机访问时在实验组，用iPad访问时在对照组。这导致用户体验不一致（如推荐算法不同），且用户行为数据被"拆分"到两个组，污染实验估计。你现在面临选择：
- 选项A：强制所有设备共享同一分组（需跨设备同步状态，工程复杂）
- 选项B：按设备ID分流而非用户ID（牺牲用户级一致性，但工程简单）
- 选项C：接受跨设备不一致，但在分析时标记"多设备用户"并做敏感性分析

你会怎么选？判断标准是什么？

## Q5: 50个实验同时在线，推荐实验和广告实验层间"行为级耦合"——正交分流够吗？ [L5 综合层]

某内容平台采用5层实验架构（UI层、推荐层、广告层、运营层、基础层），理论上层间正交。但深入分析发现：推荐层和广告层虽使用不同种子，由于用户行为链路的依赖性（推荐结果影响广告曝光），层间存在"行为级耦合"。某推荐实验和广告实验同时运行时，推荐实验的"提升点击率"效应有0.3pp被广告层的"增加广告加载"效应抵消。你面临决策困境：
- 选项1：限制同时运行的实验组合（如推荐和广告不同时跑），但牺牲实验吞吐量
- 选项2：在分析层显式建模交互效应，但增加分析复杂度
- 选项3：接受隐性干扰存在，在实验报告中声明"独立效应假设可能不成立"

你会怎么选？判断标准是什么？
