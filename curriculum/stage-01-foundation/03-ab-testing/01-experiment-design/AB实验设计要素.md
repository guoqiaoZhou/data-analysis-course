# AB实验设计要素

## 问题引入

假设你是一家内容平台的数据分析师。产品团队开发了一套新的推荐算法，想上线验证效果。PM问你："我们直接切10%流量给新算法，跑一周看数据，可以吗？"

你本能地摇头。"直接切流量跑一周"听起来简单，但里面藏着十几个可能让结论作废的陷阱：10%流量里有多少用户？这些用户是随机分配的吗？核心指标看什么——点击率、停留时长、还是完播率？如果点击率涨了但停留时长跌了，怎么决策？实验跑一周会不会太短，novelty效应还没消退？对照组用户会不会通过社交分享间接受到新算法影响（溢出效应）？

AB实验是因果推断的"黄金标准"，但黄金标准的含金量取决于实验设计的好坏。一个设计粗糙的实验，可能得出比观察性研究更误导的结论——因为人们天然更信任"实验"，对实验结果的质疑更少。在[[stage-01-foundation/01-causal-inference-basics/02-randomized-experiments/随机化实验原理|随机化实验原理]]中，我们知道随机化能创造可比的处理组和对照组；但随机化只是起点，实验设计决定了你能否在真实业务的复杂环境中，保护住随机化的成果。

实验设计涉及一连串决策：随机化单元选用户ID还是会话ID？核心指标和护栏指标各定几个？实验周期覆盖几天才能看到稳定效应？样本量够不够检测业务关心的最小效应？处理组和对照组的比例是50/50还是10/90？每个决策都有统计和业务的双重考量，没有标准答案，只有"适合当前场景"的答案。

更深层的问题是：实验不是免费的。工程开发、数据埋点、监控报警、QA测试、机会成本——这些隐性成本可能远超实验本身。一个设计不当的实验，不仅浪费资源，还可能因为"看起来显著"的错误结论，导致错误的全量决策，造成更大的损失。

## 基本原理

AB实验（随机对照实验）是因果推断的黄金标准。通过随机将用户分配到处理组（A）和对照组（B），确保两组在期望上仅差一个"是否接受处理"，从而任何结果差异都可归因于处理效应。

**核心逻辑：**
$$E[Y(1)] - E[Y(0)] = E[Y|D=1] - E[Y|D=0]$$

等式成立的条件：随机化。

## 使用场景

| 场景 | 说明 | 示例 |
|---|---|---|
| 新功能上线 | 验证功能是否提升核心指标 | 新推荐算法 vs 旧算法 |
| 策略优化 | 比较不同策略的效果 | 满减门槛200 vs 300 |
| UI改版 | 评估界面变化对用户行为的影响 | 新首页布局 vs 旧布局 |
| 运营活动 | 验证活动ROI | 发券 vs 不发券 |

## 局限性

1. **伦理限制**：某些处理不能随机化（如定价、裁员）
2. **样本量约束**：小样本或低频场景难以达到检验力
3. **溢出效应**：处理组影响对照组（社交网络、市场）
4. **长期效应难测**：实验周期有限，长期效应未知
5. **多重检验**：多个指标增加假阳性风险
6. **实施成本**：工程开发、数据埋点、监控成本

## 同类算法对比

| 方法 | 适用条件 | 优势 | 劣势 |
|---|---|---|---|
| **AB实验** | 可随机化，样本充足 | 因果推断最强 | 成本高，周期长 |
| **准实验（DID/PSM）** | 无法随机化 | 利用观察数据 | 假设强，偏误风险 |
| **反转实验** | 已全量策略 | 估计存量价值 | 撤策略有成本/风险 |
| **用户调查** | 态度/满意度 | 直接获取主观反馈 | 社会期望偏误，样本偏差 |
| **自然实验** | 外部事件造成近似随机化 | 无实施成本 | 罕见，控制有限 |

## 真实案例

### 案例1：推荐算法迭代（C端在线实验）

**背景：** 推荐团队开发新模型，在线AB实验验证效果。

**设计：**
- 随机化单元：用户ID（确保同一用户始终看到同一版本）
- 分流比例：50% / 50%
- 实验周期：2周（覆盖完整行为周期）
- 核心指标：人均点击次数（CTR）
- 护栏指标：人均浏览时长（确保不牺牲深度）

**结果：**
- CTR +3.5%（p<0.001），浏览时长 -1.2%（p=0.08，不显著）
- 决策：核心指标显著正向，护栏指标无显著负向，可全量

**注意：** 实验期间监控异常，发现某天CTR异常高，排查为某渠道用户涌入，排除后重新分析。

### 案例2：优惠券策略测试（C端离线+在线结合）

**背景：** 评估不同优惠券面额对首单转化的影响。

**设计：**
- 分层随机化：按用户价值分层（高/中/低），每层内随机
- 处理组：5元券 / 10元券 / 20元券
- 对照组：无券
- 样本量：每组2000，基于MDE=2pp计算

**分析：**
- 5元券：转化+1.2pp（不显著）
- 10元券：转化+3.5pp（显著）
- 20元券：转化+4.1pp（显著），但ROI下降（补贴成本过高）

**决策：** 选10元券，平衡效果与成本。

### 案例3：B端功能灰度（商家运营）

**背景：** 新数据分析功能面向商家，评估使用率和对GMV的影响。

**设计：**
- 随机化单元：商家ID
- 处理组：开放功能（10%商家）
- 对照组：不开放（90%商家）
- 周期：1个月
- 指标：功能使用率、30天GMV、商家满意度

**挑战：**
- 商家数量少（总计500），样本量不足
- 商家间差异大（行业、规模）
- 处理组商家可能向对照组商家"炫耀"功能（溢出）

**调整：**
- 改为配对设计：按规模/行业配对，每对中随机分配
- 增加样本量至30%商家
- 监控商家间互动，发现溢出后缩小实验范围

## 行业使用经验

### 经验1：MDE不是"越小越好"——我们曾因MDE=0.5pp而跑了两个月实验

某大厂推荐算法实验，分析师按"能检测0.5pp变化"计算样本量，结果需要跑8周。但业务方在第4周就等不及了，强行停止分析，p=0.07（不显著），结论"无效"——实际上真实效应约1pp，若按MDE=1.5pp设计，4周就能显著。**教训：MDE必须与业务最小关注效应（Minimum Business Meaningful Effect）对齐。若业务方认为"变化<2pp不值得动"，MDE设0.5pp就是浪费资源。**

### 经验2：护栏指标"不显著负向"不等于"安全"——我们曾漏掉了一个-5%的用户体验指标

美团某实验核心指标CTR+5%（显著），护栏指标"页面停留时长"-3%（p=0.12，不显著）。PM据此全量。上线后用户投诉"推荐内容质量下降"。回头看：-3%虽然统计不显著，但效应方向明确为负，且样本量足够（检验力>80%）——不显著只是因为效应量小，不代表"无害"。**规范：护栏指标必须同时看效应方向和置信区间。若置信区间包含"不可接受的负面效应"（如-5%），即使不显著也需警惕。**

### 经验3："全量效应≠实验效应"——小流量实验的陷阱

滴滴某定价实验在小流量（5%用户）下显示GMV+3%（显著），全量后仅+1%。原因在于：小流量用户多为价格敏感型（低价值用户更容易被实验策略影响），全量后高价值用户响应弱，效应被稀释。**建议：实验设计阶段就考虑"全量推广后的效应衰减"，在分析报告中明确声明"实验估计为上界，全量后预期效应约为实验效应的X%"。**

### 经验4：分层随机化的"层"不是越多越好——我们曾分了20层，结果每层样本量不足

某B端功能按"行业×规模×城市"分了20层，每层内随机化。结果核心层（如"餐饮-大型-一线城市"）样本充足，但边缘层（如"零售-小型-三线城市"）每组仅20个商家，统计功效几乎为零。**规则：分层数应控制在"核心协变量数×2-3"以内，且每层处理组和对照组至少各100个样本。分层是为了平衡，不是为了展示分类能力。**

## 代码示例

```python
import numpy as np
import matplotlib.pyplot as plt
from scipy import stats

np.random.seed(42)

# MDE 与样本量计算
def sample_size_per_group(baseline_mean, mde, std, alpha=0.05, power=0.8):
    z_alpha = stats.norm.ppf(1 - alpha / 2)
    z_beta = stats.norm.ppf(power)
    n = 2 * (z_alpha + z_beta) ** 2 * (std ** 2) / (mde ** 2)
    return int(np.ceil(n))

# Power 曲线模拟
def simulate_power_curve(n_per_group=500, baseline_mean=100.0, std=20.0):
    effect_list = np.linspace(0, 10, 21)
    powers = []
    for effect in effect_list:
        ctrl = np.random.normal(baseline_mean, std, n_per_group)
        treat = np.random.normal(baseline_mean + effect, std, n_per_group)
        _, p = stats.ttest_ind(ctrl, treat)
        powers.append(p < 0.05)
    return effect_list, powers
```
