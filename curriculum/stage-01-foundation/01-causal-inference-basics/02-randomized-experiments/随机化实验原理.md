# 随机化实验原理

## 问题引入

假设你是某电商平台的数据分析师。运营团队设计了一个"满199减20"的活动，上线后发现活动期间订单量比活动前增长了15%。运营负责人兴奋地说："活动效果太好了，下个月全量推广！"

你皱了皱眉。这15%的增长，真的是"满减活动"带来的吗？活动期间恰逢国庆长假，用户本来就有更多时间购物；竞争对手同期没有大促，流量自然向你倾斜；天气变凉，换季需求本身就在上升。这些"时间趋势""竞争环境""季节因素"混在一起，你怎么知道订单增长里有多少是活动的功劳，有多少是"即使没有活动也会发生"的？

更隐蔽的问题是：参与活动的用户和没参与活动的用户，可能本来就不是同一群人。高活跃用户更容易看到活动入口，价格敏感型用户更可能被优惠吸引——你看到的"活动组vs非活动组"差异，可能只是在比较两类不同的人，而不是同一类人在"有活动"和"没活动"两种状态下的表现。

在[[stage-01-foundation/01-causal-inference-basics/01-potential-outcomes/潜在结果框架|潜在结果框架]]中，我们知道因果推断的根本难题是"反事实不可观测"。那么，有没有一种方法，能让我们在不同时空下创造出"可比"的处理组和对照组？答案是：随机化。如果把用户像抛硬币一样随机分成两组，一组看到活动、一组看不到，那么两组用户在期望上是完全相同的——任何后续差异都可以归因于"是否看到活动"。

但"随机化"三个字说起来简单，做起来却处处是坑。怎么抛这枚"硬币"才能保证公平？小样本下随机分组会不会"碰巧"把高价值用户都分到一组？如果处理有溢出效应（比如实验组用户把优惠分享给对照组），随机化还能保护结论吗？这些问题，就是随机化实验原理要系统回答的。

## 为什么随机化能解决因果问题

### 核心逻辑

随机化分配处理 $D$ 使得：

$$D \perp (Y(1), Y(0))$$

即处理分配与潜在结果独立。这意味着：

$$E[Y(1) | D=1] = E[Y(1)] \quad \text{和} \quad E[Y(0) | D=0] = E[Y(0)]$$

因此，简单均值差异就是 ATE 的无偏估计：

$$\hat{\tau} = \bar{Y}_{treat} - \bar{Y}_{control} \xrightarrow{p} \tau_{ATE}$$

## 随机化的类型

### 1. 完全随机化（Complete Randomization）

- 从 $N$ 个单位中随机选择 $n$ 个接受处理
- 每个分配方案等概率
- **优点**：简单；保证协变量平衡（大样本）
- **缺点**：小样本可能出现协变量不平衡

### 2. 分层随机化（Stratified Randomization / Block Randomization）

- 按关键协变量分层，每层内随机化
- 例如：按用户活跃度（高/中/低）分层
- **优点**：保证层内平衡；提高估计精度
- **缺点**：需要预先定义分层变量

### 3. 配对随机化（Paired Randomization）

- 将相似单位配对，每对中随机分配
- **优点**：方差最小化（如果配对好）
- **缺点**：配对成本高；配对错误反而增加方差

### 4. 聚类随机化（Cluster Randomization）

- 以群体为单位随机化（如班级、社区）
- **适用**：处理有溢出效应时
- **缺点**：有效样本量减小；需要聚类稳健标准误

## 实验设计要素

| 要素 | 说明 | 决策要点 |
|---|---|---|
| **随机化单元** | 哪个层级被随机分配？ | 用户级/会话级/页面级 |
| **处理定义** | 实验组具体体验？ | 清晰、可执行、可区分 |
| **对照组** | 对照组体验？ | 通常是现状（status quo） |
| **目标人群** | 实验覆盖哪些用户？ | 全量/新用户/特定区域 |
| **核心指标** | 评估实验成功与否的指标 | 1-2个，业务最关注 |
| **护栏指标** | 确保不伤害其他指标的指标 | 用户体验/系统稳定性 |
| **实验周期** | 跑多久？ | 覆盖完整用户行为周期 |
| **样本量** | 每组多少人？ | 基于 MDE 和检验力计算 |

## 随机化后的检验

### 1. 分流验证（Assignment Check）

检验随机化是否按预期执行：

- 实验组/对照组样本比例是否符合设计
- 用户是否被正确分配到对应组
- 是否存在跨组污染（如用户切换设备）

### 2. 协变量平衡检验（Balance Check）

检验处理组和对照组在关键协变量上是否相似：

- t检验或标准化均值差异（SMD）
- SMD < 0.1 通常认为平衡良好
- 若不平衡严重，检查随机化实现是否有问题

### 3. Pre-AA 检验

检验实验前处理组和对照组的趋势是否一致：

- 若 Pre-AA 有显著差异，说明随机化失败或存在选择偏误
- 解决方案：DID 校正或重新设计实验

## 估计与推断

### 差异估计量（Difference Estimator）

$$\hat{\tau} = \bar{Y}_{treat} - \bar{Y}_{control}$$

**方差：**

$$Var(\hat{\tau}) = \frac{\sigma^2_{treat}}{n_{treat}} + \frac{\sigma^2_{control}}{n_{control}}$$

### 回归估计量（Regression Estimator）

$$Y_i = \alpha + \tau D_i + \epsilon_i$$

- $\tau$ 的 OLS 估计 = 差异估计量
- 加入协变量可提高精度（如果协变量与结果相关）

### 精确检验（Randomization Inference）

- 不依赖大样本近似
- 通过置换处理分配，构造检验统计量的零分布
- 适用于小样本或复杂设计

## 常见设计缺陷

| 缺陷 | 表现 | 后果 |
|---|---|---|
| **样本污染** | 实验组用户看到对照组版本 | 效应估计偏向下偏 |
| **选择偏误** | 非随机因素决定谁进实验组 | 混杂偏误 |
| **流失偏误** | 处理组用户流失更多 | 幸存者偏误 |
| **霍桑效应** | 用户因被实验而改变行为 | 效应估计向上偏 |
| ** novelty 效应** | 新策略短期效应≠长期效应 | 决策基于短期数据 |

## 行业使用经验

### 经验1：分层随机化不是"锦上添花"，而是小样本实验的救命稻草

美团外卖某区域实验仅覆盖2000骑手，完全随机化后实验组和对照组的"历史单量"均值差异达15%（SMD=0.35）。业务方质疑"随机化失败了"。实际上不是失败，是小样本的必然波动。改为按"历史单量三分位数"分层随机化后，层内SMD全部<0.05。**教训：当样本量<5000或处理组占比<20%时，分层随机化应作为默认配置，而非可选项。**

### 经验2：Pre-AA检验"不显著"不等于"没问题"——我们曾被2期数据坑过

某大厂评估新骑手激励政策，仅使用政策前2个月数据做Pre-AA检验，p值>0.1就宣称"随机化有效"。上线后发现实验组骑手的基础配送时效本身就比对照组快5%——那2期数据检验功效过低，根本检测不出这个差异。**规范做法：Pre-AA至少覆盖4-6个时间单位（日/周），且要同时看统计检验和SMD。2期数据的"不显著"，在统计上几乎等于"没检验"。**

### 经验3：SRM（样本比例不匹配）是实验平台的"烟雾报警器"

滴滴某实验设计为50/50分流，实际运行后发现实验组占53.2%，对照组46.8%（卡方检验p<0.001）。排查发现：实验组策略导致部分用户卸载App，而这些用户不再回传数据，造成"沉默的退出者"全部落在对照组的统计假象。**SRM一旦出现，无论核心指标多漂亮，结论都不可信。必须先排查工程实现、用户流失、跨设备切换等问题。**

### 经验4：聚类随机化的代价——我们低估了标准误

某B端功能以"商家"为单位随机化，但分析时用了用户级标准误。结果核心指标"人均GMV"p=0.03（"显著"），上报全量。实际用聚类稳健标准误（cluster at 商家级别）后，p=0.11（不显著）。问题根源：同一商家下的用户行为高度相关，有效样本量远小于名义样本量。**规则：聚类随机化实验，分析时必须用聚类标准误，否则假阳性率会飙升。**
