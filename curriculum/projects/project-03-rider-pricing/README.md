# 骑手配送费动态定价 — 因果推断实战项目

> 基于STAR法则与5W2H框架的数据分析实战

---

## 一、项目背景（Situation）

### 1.1 业务场景

某头部外卖平台面临骑手供需失衡问题：

- **雨天/高峰时段**：订单激增，骑手供给不足，准时率下降，用户投诉增加
- **偏远区域**：骑手不愿接单，订单积压，用户体验受损
- **平峰时段**：骑手闲置，运力浪费

运营团队提出动态定价方案——雨天/高峰/偏远区域自动加价，声称"加价能吸引更多骑手，提升准时率"。财务团队担忧"加价会吓跑用户，订单量下降"。CEO要求数据团队给出"每个区域×每个时段的最优价格"。

### 1.2 核心矛盾

| 团队 | 立场 | 潜在盲区 |
|------|------|----------|
| 运营 | 加价→骑手增加→准时率提升 | 忽略用户流失效应 |
| 财务 | 加价→订单量下降→GMV受损 | 忽略骑手供给弹性 |
| CEO | 要"最优价格" | 忽略效应异质性 |

**本质问题**：定价效应不是单一的，而是高度异质的——雨天加价vs晴天加价，效应完全不同。不存在一个"万能最优价"。

## 二、项目任务（Task）

### 2.1 分析目标

1. **评估动态定价对核心指标的影响**
   - 订单量（用户侧）
   - 准时率（体验侧）
   - 骑手供给（运力侧）

2. **识别异质性效应**
   - 哪些区域×时段加价有效？
   - 哪些区域×时段加价无效甚至有害？

3. **给出定价策略建议**
   - 差异化定价规则
   - ROI预期与风险提示

### 2.2 关键挑战

```
┌─────────────────────────────────────────────────────────┐
│  挑战1：不能随机定价                                     │
│  → 用户体验差 + 商业逻辑不允许                           │
│  → 无法做理想AB实验                                      │
├─────────────────────────────────────────────────────────┤
│  挑战2：效应高度异质                                     │
│  → 雨天加价 vs 晴天加价，效应完全不同                    │
│  → 需要"每个区域×每个时段"的个性化估计                   │
├─────────────────────────────────────────────────────────┤
│  挑战3：模型可能误设                                     │
│  → 价格效应可能是非线性的                                │
│  → 简单线性模型会系统性偏差                              │
├─────────────────────────────────────────────────────────┤
│  挑战4：选择偏差                                         │
│  → 雨天本来订单就少，不是因为加价                        │
│  → 需要区分"相关"与"因果"                              │
└─────────────────────────────────────────────────────────┘
```

## 三、5W2H 框架

### Why（为什么要做）
- 骑手供需失衡导致用户体验下降、平台收入损失
- 当前统一定价无法应对时空异质性
- 需要数据驱动的精细化定价策略

### What（做什么）
- 评估动态定价对订单量、准时率、骑手供给的因果效应
- 识别最优定价的区域×时段组合
- 输出可落地的定价策略建议

### Who（谁来做/为谁做）
- **执行者**：数据分析师（本项目的"你"）
- **需求方**：CEO、运营团队、财务团队
- **利益相关者**：骑手、用户、商户

### When（何时做）
- 分析周期：2周（模型训练+验证）
- **业务干扰**：CEO要求1周内出结论——时间压力是真实职场的常态

### Where（在哪里做）
- 数据范围：平台全量订单数据
- 区域维度：按行政区/商圈划分
- 时段维度：按小时粒度

### How（怎么做）
- 因果推断方法：DML + Meta-Learners + 因果森林
- 对比策略：S/T/X/R-Learner + 双重稳健估计
- 验证：交叉验证 + 稳健性检验

### How much（投入/收益）
- **数据量**：10,000+条面板数据（区域×时段×日期）
- **预期收益**：
  - 有效区域：准时率提升3-5%，骑手供给增加10-15%
  - 无效区域：避免加价导致的用户流失
  - 整体：GMV提升2-4%（仅有效区域加价）

## 五、核心方法论

本项目必须使用的因果推断工具：

| 方法 | 用途 | 为什么选它 |
|------|------|-----------|
| **因果森林分裂准则** | 自动发现异质性子群 | 不需要预设交互项，数据驱动发现 |
| **因果森林置信区间** | 效应估计精度 | 知道估计有多可靠，避免过度自信 |
| **S/T/X/R-Learner** | 对比不同建模策略 | 不同场景下最优策略不同 |
| **双重稳健学习** | 模型误设保护 | 即使一个模型错了，另一个能救 |
| **DML（高维控制）** | 控制数百个协变量 | 传统方法维度灾难 |
| **Nuisance函数估计** | 用ML估计倾向得分和结果模型 | 灵活、不预设函数形式 |
| **交叉验证策略** | 选择最优Meta-Learner | 数据驱动选择，不拍脑袋 |

## 六、业务干扰（真实职场模拟）

### 干扰1：运营团队的"行业惯例"
> "雨天必须加价30%，这是行业惯例！"

**数据真相**：加价20%和30%效果一样——边际效应递减。多出来的10%只是给用户添堵。

### 干扰2：财务团队的"一刀切"
> "所有区域都必须有正ROI！"

**数据真相**：偏远区域加价反而降低总GMV——用户流失效应 > 骑手供给效应。

### 干扰3：CEO的时间压力
> "1周内出结论！"

**数据真相**：模型训练+验证需要2周。1周出的结论可能是错的。

**你的应对**：
- 用快速探索性分析给出初步方向（1周）
- 明确告知完整结论需要2周
- 区分"方向性结论"和"精确估计"

## 七、失败路径（必须经历）

### 失败1：简单回归陷阱
```
订单量 ~ β₀ + β₁·价格 + ε
```
**问题**：忽略选择偏差——雨天本来订单就少，不是因为加价。
**后果**：高估价格负面效应，得出"不能加价"的错误结论。

### 失败2：AB实验幻想
```
"我们随机给不同区域定价，看效果"
```
**问题**：不可能随机定价——用户体验差 + 商业逻辑不允许。
**后果**：实验设计不可行，项目卡壳。

### 失败3：PSM维度灾难
```
用PSM匹配"加价区域"和"不加价区域"
```
**问题**：区域×时段组合太多，高维匹配困难，匹配质量差。
**后果**：估计偏差大，结论不可靠。

**学习价值**：
- 每个失败都是真实职场会犯的错
- 展示"为什么错"比"正确答案"更重要
- 从失败中引出正确方法的必要性
