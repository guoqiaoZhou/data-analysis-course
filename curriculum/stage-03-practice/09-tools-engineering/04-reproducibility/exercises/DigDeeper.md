# 深挖 - 分析可复现性

> 对应知识点：[[stage-03-practice/09-tools-engineering/04-reproducibility/分析可复现性|分析可复现性]]

### Q1 追问链: 数据Lineage追踪为什么比代码Lineage更难实现？

**追问1:** 代码Lineage和数据Lineage的区别在哪？
**Answer:** 代码Lineage追踪"谁修改了哪行代码、什么时候、为什么"——Git完美解决这个问题。数据Lineage追踪"这个数据字段从哪些源表来、经过了哪些转换、被哪些下游使用"——涉及SQL解析、ETL管道追踪、跨系统数据流动。代码是"静态文本"，数据是"动态流"——数据在管道中流动时被清洗、聚合、关联、过滤，每一步都改变其语义和结构。追踪这个"语义演变"需要解析SQL、理解业务逻辑、映射字段级依赖，远复杂于代码的文本级diff。

**追问2:** 为什么"字段级Lineage"比"表级Lineage"更难但更有价值？
**Answer:** 表级Lineage只追踪"表A → 表B"的依赖关系，无法回答"表B的'用户活跃度'字段来自表A的哪些字段？"。字段级Lineage需要解析SQL的SELECT子句、JOIN条件、CASE WHEN逻辑、聚合函数——技术上需要SQL AST（抽象语法树）解析和业务语义理解。但字段级Lineage是"可复现性"的关键：当分析师质疑"这个指标为什么变了"，表级Lineage只能说"源表变了"，字段级Lineage能精确指出"源表的'登录次数'字段在3月15日的ETL中被修改了过滤条件，导致下游'活跃用户'口径变化"。

**追问3:** 从图数据库角度，数据Lineage的"图结构"有什么特殊性质？
**Answer:** 数据Lineage图是"有向无环图"（DAG）——数据从源表流向下游，不允许循环（否则陷入无限递归）。但真实数据管道中，"逻辑循环"常见：表A依赖表B，表B依赖表C，表C又通过某种间接方式依赖表A（如通过业务规则而非直接SQL引用）。这导致Lineage图可能出现"伪环"，需要"业务语义去环"而非简单的拓扑排序。此外，数据Lineage图是"超图"——一个字段可能来自多个源字段的聚合（如"GMV = 订单金额 × 数量 - 优惠"），传统图数据库的二元边无法表达这种"多对一"的聚合关系。需要"属性图"或"超图"模型才能准确表达。

### Q2 追问链: Docker容器化为什么能解决"在我机器上能跑"的问题，但引入了新的不可复现性？

**追问1:** Docker如何解决环境差异问题？
**Answer:** Docker将代码、依赖、操作系统库打包为"镜像"，确保"相同镜像 → 相同环境 → 相同结果"。这解决了"分析师用Mac、服务器用Linux、同事用Windows"导致的包版本差异、系统库缺失、路径分隔符不同等问题。Docker的"基础设施即代码"理念，将"隐式环境假设"转化为"显式镜像定义"。

**追问2:** Docker引入了哪些新的不可复现性风险？
**Answer:** 三个风险：① "镜像漂移"——Dockerfile中用了`apt-get update`或`pip install latest`，导致不同时间构建的镜像包含不同版本的系统包；② "基础镜像变更"——`FROM python:3.9`在官方仓库更新后，底层操作系统补丁变化；③ "构建上下文差异"——.dockerignore配置不当导致不同机器构建时包含/排除的文件不同。这些风险使得"相同Dockerfile"不保证"相同镜像"——必须用`docker build --no-cache`并锁定所有依赖的精确版本（如`pip install pandas==1.3.0`而非`pip install pandas`）。

**追问3:** 从"可复现性层次"看，Docker属于哪一层？它之上还有什么？
**Answer:** Docker解决"计算可复现性"中的"环境锁定"问题，属于第一层。但它不解决：① "数据可复现性"——Docker镜像不包含数据，数据版本仍需DVC管理；② "语义可复现性"——Docker确保"代码能跑"，但不确保"指标口径一致"（如"活跃用户"定义变化）；③ "知识可复现性"——Docker不记录"为什么用这个模型、这个参数、这个假设"。更高层次的可复现性需要"指标仓库"（语义级）、"分析知识库"（决策逻辑级）、"因果假设声明"（科学级）。Docker是必要但不充分的条件——它是"地板"，不是"天花板"。
