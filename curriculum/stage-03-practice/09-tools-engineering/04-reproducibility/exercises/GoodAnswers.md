### Q1: 季度报告结论与原版存在15%差异，为什么无法复现？

**Hints:**
- 思考"可复现性"的三个层次：计算可复现、统计可复现、科学可复现。本次问题出在哪一层？
- 回顾"计算可复现"的核心要素：代码版本、数据版本、环境版本、参数显性化。

**Answer:**
无法复现是因为缺失了计算可复现性的四个核心要素：代码版本、数据版本、环境版本、参数显性化。

- **代码无版本控制**：不知道原版用的是哪个分支、哪个commit。可能原版在分析过程中修改了代码但未记录，新分析师重写的代码逻辑有细微差异。
- **数据来源不明**：中间表已下线，新分析师只能重新从原始数据计算，但原始数据的清洗逻辑、过滤条件、关联方式未知。"同样的SQL"可能因执行时间不同而读取了不同的数据分区。
- **环境未锁定**：Python包版本未知——原版可能用了pandas 1.3（默认行为A），新分析师用了pandas 1.5（默认行为B），导致同一行代码输出不同。
- **指标口径模糊**："活跃用户"没有明确定义，原版可能是"启动App"，新分析师理解为"完成核心操作"，15%差异很大程度上来自口径差异。
- **决策规则**：任何分析项目必须满足"可复现三件套"：① Git版本控制的代码；② 依赖锁定文件（requirements.txt/poetry.lock）；③ 数据来源与转换的lineage说明。对于关键报告（如季度董事会报告），额外要求④ 数据快照归档（分析所用数据的精确副本）；⑤ 参数配置文件（所有影响结果的参数显性化，禁止硬编码）。

### Q2: "Git管理代码"和"Git管理数据"为什么前者是最佳实践、后者是反模式？

**Hints:**
- 思考Git的设计目标：版本控制文本文件（代码），还是二进制大文件（数据）？
- 回顾"仓库膨胀"问题：数据文件提交Git后，克隆和diff操作会发生什么？

**Answer:**
Git的设计目标是小文本文件（代码），数据文件需要用专门工具管理。

- **Git管理数据的问题**：
  - **仓库膨胀**：一个1GB的CSV提交后，Git仓库永远保留该文件的历史版本（即使后续删除），克隆时下载全部历史，仓库体积失控。
  - **diff无用**：Git的diff算法针对文本行优化，对二进制数据（如Parquet、模型文件）无法生成有意义的diff，版本对比无意义。
  - **性能崩溃**：Git对大量小文件的索引效率低，百万级数据文件会导致git status耗时数分钟。
- **数据的正确管理方式**：
  - **对象存储**（S3/OSS）+ **DVC**：数据文件存储在对象存储中，Git只管理一个轻量的".dvc"指针文件（记录数据的哈希和存储路径）。DVC负责数据的版本控制、缓存、远程同步，与Git的工作流类似但针对大文件优化。
  - **数据快照**：对于关键分析，在分析执行时将所用数据导出为快照（如按日期命名的数据分区），快照路径记录在分析代码中。
- **决策规则**：
  - **代码**：必须用Git管理，包括分析脚本、SQL、配置文件。
  - **数据**：禁止直接提交Git。小数据（<10MB的配置表）可例外，但需团队共识；大数据必须用DVC或对象存储管理。
  - **模型文件**：与数据同等对待，用DVC管理。

### Q3: 为什么客户健康度模型同一批客户不同时间得分不同？

**Hints:**
- 思考"隐式参数"的危害：哪些常见的"当前状态"会被无意识硬编码？
- 回顾"分析时间截断"（As-of-date Analysis）：如何确保分析不使用"未来信息"？

**Answer:**
这是典型的"隐式参数"和"未来信息污染"问题——"当前日期"被无意识硬编码，导致历史分析被未来数据污染。

- **错误根源**：特征工程代码中使用了`datetime.now()`或`CURRENT_DATE`等隐式参数，而非显式的"分析基准日期"。例如，计算"过去30天登录次数"时，如果代码在2024年6月运行，"过去30天"是5月；如果在2024年7月运行，"过去30天"是6月——同一客户的行为数据不同，得分自然不同。
- **更隐蔽的污染**：如果模型训练时用了"当前日期"，而推理时也用了"当前日期"，则训练数据和推理数据的时间窗口不一致，导致模型性能漂移。更严重的是，如果"当前日期"在训练时包含了"未来信息"（如用2024年6月的"当前日期"训练，但部分客户数据实际来自7月），则模型在训练时"偷看了"未来，评估指标虚高。
- **为什么总是犯这个错**：`datetime.now()`写起来方便，分析师不会意识到它在不同时间运行结果不同。只有在做"历史回测"或"模型复现"时，问题才会暴露。
- **修正方法**：
  - **参数显性化**：所有时间相关参数（基准日期、时间窗口、预测 horizon）必须提取为配置文件或命令行参数，禁止硬编码`datetime.now()`。
  - **分析时间截断**：历史分析必须模拟"当时已知的信息"——如分析2024年1月的客户健康度，只能用2024年1月及之前的数据，不能用2月的数据。
  - **回归测试**：对核心模型建立"回归测试套件"——固定输入数据和参数，验证输出得分是否稳定。得分变化时自动报警。

### Q4: Jupyter Notebook为什么不应作为"最终交付物"？

**Hints:**
- 思考Notebook的"交互性"与"可维护性"的权衡：探索时灵活，生产时脆弱。
- 回顾"Notebook模块化"的最佳实践：如何将探索性代码转化为可维护的生产代码？

**Answer:**
Notebook是"草稿纸"，适合探索但不适合生产，因为存在四个结构性问题。

- **执行顺序混乱**：Notebook允许非线性执行（先跑单元格5、再跑单元格2），导致变量状态不一致。新人接手时，"从头运行"可能因顺序错误而失败，"按原顺序运行"又难以还原。
- **代码与输出混杂**：图表、表格、Markdown说明散落在代码中间，难以提取核心逻辑。"这个图表是哪段代码生成的？"需要人工追踪。
- **无法自动化**：Notebook需要手动逐个单元格点击，无法嵌入CI/CD流水线或定时任务。月报自动化时，Notebook成为瓶颈。
- **测试困难**：Notebook中的代码难以写单元测试——函数定义和调用散落在不同单元格，没有明确的模块边界。
- **平衡方案**：
  - **探索阶段**：用Notebook自由探索，快速迭代。
  - **交付阶段**：将核心逻辑提取为模块化Python/R脚本（函数/类），配测试用例，再用Notebook调用并展示结果。Notebook变成"展示层"，脚本变成"逻辑层"。
  - **规范**：团队制定"Notebook交付标准"——交付的Notebook必须满足：① 可从头线性运行（Kernel → Restart & Run All 通过）；② 核心逻辑已提取为import的模块；③ 包含README说明运行步骤和依赖。
- **决策规则**：
  - **如果分析是一次性探索**（如"看看这个数据长什么样"），Notebook可交付，但需注明"探索性，不可复用"。
  - **如果分析是周期性报告**（如月报、季报），必须模块化脚本 + Notebook展示，禁止纯Notebook交付。
  - **如果分析是核心模型/策略**，必须模块化 + 测试 + 版本控制，Notebook仅用于演示。

### Q5: 小分析项目是否"过度工程"？如何在"足够好"和"完美"之间平衡？

**Hints:**
- 思考"风险分级"：不同分析项目的"错误成本"差异有多大？
- 回顾"成本-收益"权衡：严格的可复现性投入，在什么情况下ROI最高？

**Answer:**
可复现性应该"分层"——根据分析项目的影响范围和错误成本，匹配不同级别的可复现性标准。

- **分层标准**：
  - **L1 临时取数（10分钟-2小时）**：口头或Slack交付，无需版本控制。但要求：① SQL保存到共享查询平台（如Metabase、Superset）；② 结果截图标注"临时数据，未验证"。
  - **L2 常规分析（半天-3天）**：Git管理代码 + requirements.txt锁定依赖 + 数据来源说明。适用于周报、专项分析、非关键决策支持。
  - **L3 关键报告（1周-2周）**：L2 + 数据快照归档 + 参数配置文件 + Notebook模块化。适用于季度复盘、预算申请、战略汇报。
  - **L4 核心模型/策略（长期维护）**：L3 + Docker容器化 + DVC数据版本控制 + 指标仓库引用 + CI/CD自动化。适用于风控模型、推荐策略、定价算法等直接影响业务的核心系统。
- **决策规则**：
  - **错误成本分级**：如果分析错误会导致"百万级损失"或"战略方向错误"，必须L4；如果错误只会导致"方向微调"，L2或L3即可。
  - **复用频率分级**：如果分析代码需要"每月复用"（如月报），必须至少L3（模块化+自动化）；如果"一次性使用"，L2足够。
  - **审计要求分级**：金融、医疗等受监管行业，即使小分析也需L3（可追溯）；互联网行业可适度放宽。
  - **团队共识**：与团队明确分层标准和升级路径——"当前项目定为L2，但如果结论被质疑，升级到L3重做"。避免"一刀切"的过度工程，也避免"一刀切"的草率交付。
