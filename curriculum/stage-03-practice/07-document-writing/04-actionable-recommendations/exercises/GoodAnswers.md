# 好问练习解答 - 落地建议颗粒度

> 对应知识点：[[stage-03-practice/07-document-writing/04-actionable-recommendations/落地建议颗粒度|落地建议颗粒度]]

---

### Q1: 为什么"建议优化购物车页面"等于什么都没说？

**Hints:**
- 思考"优化"这个词的含义——它指向一个"方向"还是"动作"？
- 回忆SMART原则中的"Specific"和"Assignable"

**Answer:**
因为"优化"是一个方向，不是动作。执行团队拿到"优化购物车页面"，不知道改什么、谁改、什么时候改完、改完怎么评估。这等于把分析的价值停在"发现问题"，没有完成"推动行动"的最后一公里。

好的落地建议应满足SMART原则的变体：
- **Specific**：明确做什么，不模糊
- **Measurable**：有预期效果，可评估
- **Assignable**：能明确责任方
- **Realistic**：基于资源约束，可执行
- **Time-bound**：有明确时间线

决策规则：
- **如果建议中缺少"做什么"的具体描述** → 太粗，执行团队无抓手
- **如果建议中缺少"Owner"** → 无人负责，建议沦为"待办清单上的幽灵任务"
- **如果建议中缺少"时间节点"** → 无限期拖延，建议失去时效性
- **如果建议中缺少"验收标准"** → 无法评估效果，建议成败无人知晓

大厂实践：美团内部要求落地建议必须包含"Owner + 时间节点 + 验收标准"三个要素，缺一不可。

### Q2: "改按钮颜色"和"优化页面"哪个更错？

**Hints:**
- 思考分析师的角色边界——是"问题诊断者"还是"方案设计者"？
- 回忆"GPS导航"比喻：告诉你方向和大致距离，但不教你怎么踩油门

**Answer:**
两者都错，但错在不同维度。"优化页面"错在"太粗"，没有行动抓手；"改按钮颜色"错在"太细"，侵入执行团队的决策空间。

"刚刚好"的颗粒度是：**明确方向 + 可验证预期 + 留给执行团队的设计空间**。

示例对比：
- ❌ 太粗："建议优化购物车页面，减少用户流失"
- ❌ 太细："建议将按钮颜色从#1890FF改为#096DD9，字体大小从14px改为16px"
- ✅ 适中："建议简化结算页流程，将3步合并为2步，预计减少15%流失"

"适中"的建议包含：
1. **方向明确**：简化结算页流程
2. **范围清晰**：3步合并为2步（不是"优化整个页面"）
3. **预期效果**：预计减少15%流失（可验证）
4. **设计空间**：具体怎么合并、UI如何调整——留给产品和设计团队

决策规则：
- **如果建议涉及具体设计元素（颜色、字体、像素）** → 太细，退回一级抽象
- **如果建议只涉及抽象方向（优化、提升、改善）** → 太粗，前进一步具体化
- **如果建议涉及"功能/流程/策略层面的改变"** → 颗粒度适中

### Q3: 为什么分析师总是忘记写"谁来做"和"什么时候做完"？

**Hints:**
- 思考"分析师角色边界"是一个"客观分工"还是"主观选择"
- 回忆美团要求的"Owner + 时间节点 + 验收标准"

**Answer:**
这是**角色认知偏差**，不是客观分工限制。

分析师的完整价值链条是："发现问题 → 分析原因 → 给出建议 → 推动行动 → 评估效果"。如果只做到"给出建议"但不管"谁来做、什么时候做、怎么评估"，分析的价值只完成了一半。

为什么分析师倾向于回避：
1. **安全感边界**："我只分析不推荐"是一种防御机制——推荐错了可以推卸责任
2. **组织惯性**：部分公司确实将分析师定位为"纯支持角色"，不鼓励介入执行
3. **能力盲区**：部分分析师不熟悉业务执行流程，不知道建议该由谁执行

为什么大厂要求必须写：
- 没有Owner的建议 = 没有执行力的建议 = 没有价值的建议
- 没有时间节点的建议 = 无限期拖延 = 建议失去时效性
- 没有验收标准的建议 = 无法闭环 = 无法积累"分析→行动→效果"的经验

正确做法：
- **Owner**：建议到团队即可，不需要到个人。例如"产品团队负责""运营团队负责"
- **时间节点**：区分短期/中期/长期。例如"本周""本月""本季度"
- **验收标准**：可量化的效果指标。例如"预计减少15%流失""目标提升5%留存"

决策规则：
- **如果分析师不知道Owner是谁** → 去问，或写"建议由XX团队评估可行性后确定Owner"
- **如果分析师不确定时间节点** → 写"建议在XX时间窗口内启动，具体排期由执行团队确定"
- **如果分析师无法量化验收标准** → 写"建议以XX指标为评估维度，具体目标由执行团队设定"

即使不能精确，也要给出框架——空白等于放弃。

### Q4: 历史数据的外推效果，该怎么写"预期效果"？

**Hints:**
- 思考"相关性"和"因果性"在建议中的表达方式
- 回忆"建议"与"假设"的区分

**Answer:**
**明确区分"有数据支撑的建议"和"待验证假设"，并用不同方式标注。**

有数据支撑的建议：
> "将客服响应时间从2小时缩短至30分钟，预计提升满意度10%（基于历史相关性分析：过去三个月响应时间缩短群体的满意度较未缩短群体高10pp，样本N=5000）"

待验证假设：
> "用户流失可能是因为价格敏感（当前仅为推测，基于流失用户画像中价格敏感标签占比偏高），建议下周做价格弹性测试验证"

关键区别：
- **建议**：基于历史数据的因果推断或强相关性，有明确的数据支撑
- **假设**：基于推测或弱相关性，需要后续验证

决策规则：
- **如果历史数据支持因果推断**（如A/B测试结果） → 直接写预期效果，标注数据来源
- **如果历史数据只支持相关性**（如观察性数据） → 写预期效果，但标注"基于历史相关性，非严格因果推断"
- **如果预期效果基于推测**（无直接数据支撑） → 写为"待验证假设"，并给出验证方法

为什么必须写预期效果：
- 没有预期效果，建议无法评估，等于"做了不知道有没有用"
- 即使预期效果基于相关性而非因果性，标注清楚后，读者可以自行判断可信度
- "过度自信"的反面不是"不写预期效果"，而是"诚实标注预期效果的依据和不确定性"

### Q5: 主方案和备选方案，决策者问"你到底推荐哪个"，怎么回答？

**Hints:**
- 思考"提供备选方案"的初衷——是"降低决策风险"还是"逃避判断责任"
- 回忆阿里内部要求：涉及预算超过50万的建议，必须提供至少一个低成本验证方案

**Answer:**
**备选方案的目的是降低决策风险，不是逃避判断责任。** 如果分析师不提供明确推荐，等于把"分析"降格为"信息罗列"，失去了分析师的核心价值。

正确回应方式：
> "基于当前数据，我推荐**备选方案（灰度测试）**作为第一步。理由：①主方案投入大且效果不确定性高，历史数据未验证过类似改动的效果；②灰度测试可以在两周内以10%成本验证核心假设，若效果显著再推进主方案；③即使灰度测试效果不及预期，损失可控，且能获得数据反馈优化主方案设计。如果业务方对时间窗口有强要求、无法接受两周验证期，则选择主方案，但需同步准备回滚机制。"

这个回应包含：
1. **明确推荐**：不是"两个都行"，而是"基于XX条件，我推荐XX"
2. **推荐理由**：用数据或逻辑支撑推荐
3. **条件切换**：如果条件变化（如时间压力），推荐也变化
4. **风险对冲**：即使推荐主方案，也准备回滚机制

决策规则：
- **如果主方案有充分数据支撑（如A/B测试验证过）** → 推荐主方案，备选方案作为风险对冲
- **如果主方案缺乏验证、风险高** → 推荐备选方案（验证后再决策），并说明验证通过后的升级路径
- **如果决策者拒绝承担任何风险、要求"安全选项"** → 推荐备选方案，但明确标注"保守选择可能错失机会成本"
- **如果决策者要求"最大收益"、愿意承担风险** → 推荐主方案，但明确标注风险点和监控指标

关键原则：分析师的价值不是"给选项"，而是"在不完备信息中做出基于数据的判断"。不提供推荐 = 放弃分析师的核心职责。
