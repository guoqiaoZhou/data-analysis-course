# 好问练习解答 - 核心结论写作

> 对应知识点：[[stage-03-practice/07-document-writing/02-conclusion-writing/核心结论写作|核心结论写作]]

---

### Q1: 为什么"本月新客获取成本为150元"不是结论？

**Hints:**
- 思考"事实陈述"和"判断"的本质区别——前者是"发生了什么"，后者是"为什么会这样"+"意味着什么"
- 回忆美团内部报告模板要求的"So What"原则

**Answer:**
因为结论必须是**判断**，而非单纯的事实陈述。"150元"是数据，任何人看报表都能得到。但"环比上升20%"是变化判断，"主因是信息流渠道竞价加剧"是归因判断，"建议下周测试短视频自然流量渠道"是行动判断。三者合起来，才是业务方 hire 分析师的目的。

核心结论的提炼公式：
> **核心结论 = 核心发现（数据事实） + 原因判断（归因分析） + 行动建议（决策方向）**

决策规则：
- **如果结论只包含"数据事实"** → 不是结论，是数据呈现。必须补充"So What"
- **如果结论包含事实+判断但无行动** → 是半成品。业务方会问"所以我们要做什么？"
- **如果结论包含事实+判断+行动** → 完整结论。但行动建议的颗粒度需参考[[stage-03-practice/07-document-writing/04-actionable-recommendations/落地建议颗粒度|落地建议颗粒度]]

大厂实践：美团内部数据分析报告模板要求结论栏必须包含"So What"——即"这意味着什么，我们要做什么"。纯描述性结论会被打回重写。

### Q2: "核心结论先行"和"渐进式揭示"到底谁对？

**Hints:**
- 对比两者的适用条件：一个适合"结论明确"，一个适合"结论复杂"
- 思考你的读者是"需要快速决策"还是"需要先理解复杂性"

**Answer:**
两者不是对错关系，是**适用场景不同**。

决策规则：
- **如果结论明确、需快速决策** → 核心结论先行。例如："建议全量上线新版推荐策略"——结论清晰，不需要铺垫。
- **如果议题复杂、结论涉及多维度权衡** → 渐进式揭示或分层结论。例如："创作者生态呈现'头部稳定、腰部流失、底部涌入'的分化趋势"——单一结论无法概括，需要分层展开。
- **如果读者是高层决策者、时间稀缺** → 结论先行，复杂内容放附录。
- **如果读者是技术评审、需要过程可追溯** → 正文可用渐进式，但首页仍须放"执行摘要"（结论先行）。

关键判断标准：
- **结论的确定性**：如果数据充分支撑一个明确判断 → 结论先行
- **结论的复杂度**：如果必须分层才能准确表达 → 渐进式或分层结论
- **读者的时间成本**：时间越稀缺 → 越需要结论先行

阿里内部要求："一页纸一个观点"，超过一个核心结论必须拆页。这是对"结论先行"的强化执行。

### Q3: 为什么分析师总用"可能的原因有5个"来逃避下判断？

**Hints:**
- 思考"下判断"对分析师意味着什么——是能力问题还是心理安全的问题？
- 回忆CEO的反馈："即使判断有主观成分，也要基于数据权重给出最可能的结论"

**Answer:**
这是分析师的**职业心理陷阱**，不是能力问题。

根本原因：
1. **逃避决策责任**：下判断意味着可能出错，而"可能性列表"把决策责任推回给读者（"你自己选"）。
2. **数据完美主义**：分析师希望100%确定才下结论，但业务决策在信息不完备下进行是常态。
3. **组织安全感缺失**：如果过去下判断出错被追责，分析师会本能地用"可能性"作为防御机制。

为什么业务方不接受"可能性列表"：
- 业务方 hire 分析师不是为了得到"可能"，而是为了得到"基于数据的判断"。
- "5个可能原因"等于要求业务方自己再做一次分析师的工作——筛选权重、判断主次。
- 即使判断有主观成分，明确给出"最可能的主因（贡献60%）+ 次要原因（贡献25%）"，也比"5个可能"有价值得多。

正确做法：
- 基于数据权重和领域知识，明确给出**判断权重**
- 标注置信度："主因判断置信度70%，建议同步验证XX假设"
- 如果确实无法判断主次，诚实写"当前数据无法判断主次，建议补充XX分析"——这也是一种有价值的结论

### Q4: 数据不充分时，该给判断还是该说"不知道"？

**Hints:**
- 思考"诚实说不知道"在业务方眼里是"无能"还是"专业"
- 回忆核心结论写作的"结论要敢于说不知道"经验

**Answer:**
**诚实说不知道，但必须附带行动方向。**

两种错误做法：
- **强行给判断** → 可能导致错误决策，且一旦出错，分析师信誉受损
- **只说"不知道"** → 没有提供任何价值，业务方觉得"那我 hire 你干嘛"

正确做法（第三种写法）：
> "当前数据无法判断转化率下降的主因是流量质量变化还是产品体验问题（诚实声明），建议下周补充用户行为路径分析后再做决策（行动方向）。"

这个结论同样提供了价值：
1. **划定了已知边界**："我知道什么不知道什么"
2. **指明了下一步**："补充XX分析"
3. **避免了错误决策**：防止基于不充分数据匆忙行动

决策规则：
- **如果数据不充分但时间紧迫、必须决策** → 给出"最可能判断+置信度+风险提示"，例如："基于现有数据，最可能主因是流量质量变化（置信度60%，非确定性判断），建议同步准备产品体验排查方案作为备选"
- **如果数据不充分且决策不紧迫** → 诚实说不知道，并给出补充分析计划
- **如果数据充分但分析师不敢下判断** → 能力/心理问题，需刻意练习

### Q5: 多个重要发现，该拆成多篇还是在一篇里分层呈现？

**Hints:**
- 思考"一个文档一个核心结论"原则的本质目的——是"信息组织"还是"决策效率"
- 回忆创作者生态案例中的"分层结论"结构

**Answer:**
判断标准是**发现之间的关联性和决策耦合度**。

决策规则：
- **如果发现之间强关联、指向同一战略方向** → 一篇报告，用分层结论。例如："创作者生态呈现分化趋势"（核心结论）+ "头部稳定/腰部流失/底部涌入"（分层支撑）。三个发现共同指向"腰部扶持"这一核心行动。
- **如果发现之间弱关联、需要不同团队分别执行** → 拆成多篇短报告。例如："头部创作者留存分析"给创作者运营团队，"腰部创作者流失分析"给产品团队，"新入驻增长分析"给增长团队。
- **如果发现之间矛盾、需要权衡取舍** → 一篇报告，但必须明确标注"本文涉及多个独立结论，优先级如下"。

阿里内部"一页纸一个观点"的深层逻辑：
- 不是禁止一篇报告有多个发现，而是禁止**核心结论模糊**
- 如果三个发现都很重要，可以写："本文核心结论：腰部创作者流失是最大风险点。补充发现：头部稳定、底部增长。"
- 关键是让读者在30秒内知道"最重要的判断是什么"

创作者生态案例的正确写法：
- **核心结论**："创作者生态呈现'头部稳定、腰部流失、底部涌入'的分化趋势，腰部创作者扶持机制失效是最大风险点"
- 这个结论没有掩盖腰部问题，而是将三个发现组织在一个有优先级的判断中
