# 好问练习解答 - 方法说明写作

> 对应知识点：[[stage-03-practice/07-document-writing/03-methods-section/方法说明写作|方法说明写作]]

---

### Q1: 为什么"模型准确率95%"不能建立可信度？

**Hints:**
- 思考"数字本身"和"数字背后的条件"哪个更重要
- 回忆案例中模型失效的两个具体原因：数据时效性和更新频率

**Answer:**
因为可信度不来自"结论数字"，而来自"结论数字是在什么条件下得出的"。95%的准确率如果基于半年前的数据、在T+7更新频率下、且未经过近期产品改版后的验证——这个数字对当前决策几乎没有参考价值。

方法说明的核心目的不是"展示技术"，而是**建立可信度**。让读者相信：这个分析师知道自己在做什么，结论不是拍脑袋出来的。

建立可信度需要回答四个问题：
1. **数据来源**：用了什么数据？时间范围？样本量？
2. **分析口径**：指标如何定义？过滤条件是什么？
3. **方法论**：用了什么分析技术？模型假设是什么？
4. **局限性**：方法本身有什么约束？结论在什么条件下成立？

决策规则：
- **如果报告缺少"数据来源+口径+方法+局限性"中任意一项** → 可信度不完整，读者有理由质疑
- **如果只给数字不给条件** → 等于说"相信我"——这不是专业沟通，是权威主义
- **如果方法说明包含了局限性** → 即使数字不漂亮，读者也会相信分析师对边界有清醒认知

### Q2: "详细方法附录"和"精简方法脚注"到底该写多详细？

**Hints:**
- 思考"方法说明的长度"是由"方法复杂度"决定，还是由"读者需求"决定？
- 回忆字节跳动的三版本实践

**Answer:**
方法说明的详细程度由**读者的技术深度和验证需求**决定，而非方法本身的复杂度。

决策规则：
- **给CTO/技术负责人** → 方法说明可以占30%，包含模型公式、参数调优过程、随机种子设置
- **给运营/业务负责人** → 方法说明控制在10%以内，用通俗语言解释关键假设和数据来源
- **给CEO/高管** → 方法说明可以是一句话"基于最近90天全量用户行为数据，采用 cohort 分析方法"，但这句话必须包含"数据来源+方法名称"两个要素

字节跳动内部实践：同一分析结果产出三个版本——"技术版"（含代码和详细方法）、"业务版"（精简方法说明）、"高管版"（几乎无方法说明，聚焦结论和建议）。

如果只能产出一份报告，采用"分层结构"：
1. **正文**：精简方法说明，控制在10%以内
2. **附录**：详细方法、公式、参数
3. **脚注**：关键口径定义（如"活跃用户定义见附录A"）

关键原则：技术读者可以深入附录，业务读者不会被正文中的技术细节干扰——各取所需。

### Q3: 为什么分析师喜欢在方法说明里堆砌复杂技术术语？

**Hints:**
- 思考"方法崇拜"和"能力展示"的心理动机
- 回忆案例中分析师答不上来"具体什么业务场景会失效"

**Answer:**
这是**方法崇拜（Method Idolatry）**——用复杂方法的"光环"来掩盖数据质量问题或结论薄弱。

根本原因：
1. **安全感替代**：复杂术语给分析师一种"我很专业"的心理安慰，即使他并不理解方法的边界
2. **防御机制**：当数据质量有瑕疵或结论不够 robust 时，复杂方法可以作为"挡箭牌"——"我用了这么高级的方法，结论应该可信"
3. **组织激励偏差**：部分公司文化将"方法复杂度"等同于"分析师能力"，导致分析师有动力炫技

为什么这是错误的：
- 方法说明的核心目的是"建立可信度"，而可信度来自"对方法边界的清醒认知"
- 一个能清楚说出"模型在什么场景下会失效"的分析师，比一个只会罗列技术术语的分析师更可信
- 美团内部要求：所有预测类报告必须在方法说明中包含"模型在什么场景下会失效"

正确做法：
- 方法说明先写"数据来源和口径"（这是大多数争议的根源）
- 再写"方法名称和核心假设"（一句话能说清的不展开）
- 最后写"局限性"（主动暴露边界，反而增加可信度）

### Q4: 给CEO看的报告，方法说明可以只有一句话吗？

**Hints:**
- 思考"最小必要方法信息"应该包含哪些要素
- 回忆案例中CEO追问"活跃用户怎么定义"的场景

**Answer:**
**可以只有一句话，但这句话必须包含两个不可省略的要素：数据来源 + 方法名称。**

高管版方法说明的"最小必要信息"：
- ✅ "基于最近90天全量用户行为数据，采用 cohort 分析方法"——包含数据来源（最近90天全量）+ 方法（cohort分析）
- ❌ "采用先进的机器学习模型"——缺少数据来源、方法名称模糊、"先进"是主观评价而非客观描述

但"一句话方法说明"有一个隐藏风险：如果CEO追问技术细节，你必须能立刻展开。这意味着：
1. 你本人必须清楚详细方法（不能是别人写的报告你照念）
2. 详细方法文档必须随时可调用（随身携带或能快速打开）
3. 一句话中提到的每个术语都必须经得起追问（如"活跃用户"的定义）

决策规则：
- **如果方法有争议历史** → 即使高管版，也要在脚注或附录中明确定义口径
- **如果结论是预测类/模型类** → 高管版必须包含"模型在什么场景下会失效"的提示
- **如果CEO过去追问过技术细节** → 提前准备一页"方法速览"作为备用幻灯片

### Q5: 主动暴露局限性，会不会让报告显得"不够有力"？

**Hints:**
- 思考"局限性"和"结论可靠性"在读者心中的关联方式
- 回忆案例中业务负责人说"有这么多限制，那还能不能用"

**Answer:**
**不会削弱说服力——但写法决定效果。**

局限性声明的错误写法（削弱说服力）：
- "本分析存在一定局限性"——泛泛而谈，等于什么都没说，读者会觉得"你在推卸责任"
- "模型基于历史数据，实际效果可能因市场环境变化而不同"——过于笼统，没有具体边界

局限性声明的正确写法（建立可信度）：
- "模型在整体样本上MAPE=8.5%，但在节假日场景下误差扩大至15%，建议节假日前采用人工修正"——具体、有数字、有行动建议

关键区别：
- **错误写法**：局限性是"免责声明"，与结论脱节，读者理解为"结论可能错了"
- **正确写法**：局限性是"有效性边界"，与结论绑定，读者理解为"结论在XX条件下成立，在YY条件下需要调整"

决策规则：
- **如果局限性是"结论可能完全错误"** → 这不是局限性，是结论本身有问题，需要重做分析
- **如果局限性是"结论在特定条件下成立"** → 明确写出条件，并给出"条件变化时的应对方案"
- **如果业务方追问"到底能不能用"** → 回答"在XX条件下可以直接用，在YY条件下需要配合ZZ措施"——给确定性判断，不是模棱两可

腾讯内部要求：局限性声明必须包含"具体什么局限"、"对结论的潜在影响"、"建议的补救措施"三个要素。这种写法将局限性从"免责声明"转化为"行动指南"。
