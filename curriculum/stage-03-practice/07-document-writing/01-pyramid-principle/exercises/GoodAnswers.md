# 好问练习解答 - 金字塔原理

> 对应知识点：[[stage-03-practice/07-document-writing/01-pyramid-principle/金字塔原理|金字塔原理]]

---

### Q1: 为什么数据分析报告不能按"数据收集→分析过程→得出结论"的顺序来写？

**Hints:**
- 思考"读者的时间成本"和"认知负荷"——老板每天要看多少份报告？
- 回忆认知心理学中的"工作记忆容量"概念，4±1个组块意味着什么

**Answer:**
因为读者的注意力是稀缺资源，不是无限容量。认知心理学研究表明，人类工作记忆容量约为4±1个组块。如果前15页都是数据铺垫，读者的工作记忆已经被无关信息填满，等你说到结论时，他已经失去了处理核心观点的认知资源。

决策规则：
- **如果读者是决策层（老板、高管）** → 必须用结论先行，30秒内给出核心观点，后续每页回答"为什么"
- **如果读者是技术评审层** → 可以在附录保留完整过程，但正文仍须结论先行
- **如果文档用于探索性讨论** → 不适用金字塔原理，改用[[stage-03-practice/07-document-writing/01-pyramid-principle/金字塔原理|金字塔原理]]中提到的"不适合探索性分析"的替代方案

核心原则：分析的价值不在于你做了多少工作，而在于读者能否在30秒内抓住你的核心观点。

### Q2: 金字塔原理和SCQA模型到底该用哪个？

**Hints:**
- 对比两者的底层目的：一个追求"效率"，一个追求"共鸣"
- 思考你的读者是"需要快速决策"还是"需要先理解问题的严重性"

**Answer:**
两者的本质差异是"效率优先"vs"共鸣优先"。

决策规则：
- **如果场景是"已知问题，需要方案"** → 用金字塔原理。结论先行，直接给判断。例如："流失率上升主因是新手引导体验差，建议立即优化注册流程。"
- **如果场景是"问题未被认知，需要先建立共识"** → 用SCQA模型。先讲情境和冲突，让读者感受到"这确实是个问题"，再给出答案。例如："新用户注册后72小时内流失率同比上升40%（情境），但产品团队认为当前 onboarding 流程已经优化过（冲突），为什么数据和产品感知不一致（疑问），因为产品团队只看完成率，没看中途跳出率（答案）。"
- **如果文档是数据密集型分析报告** → 金字塔原理。SCQA的"故事线"结构松散，不适合承载大量数据。
- **如果文档是战略咨询汇报、需要争取资源** → SCQA或故事线。需要情感共鸣和情节张力。

大厂实践：字节跳动内部汇报模板强制要求"核心观点 + 3个支撑维度"，这是金字塔原理的变体。

### Q3: 为什么塔身论据总是超过7个且彼此重叠？

**Hints:**
- MECE原则是一个"理想状态"还是"必须做到"？
- 思考"归类"和"分层"的区别——是不是该用子金字塔？

**Answer:**
MECE原则（相互独立、完全穷尽）是一个追求方向，不是绝对标准。真实业务场景边界模糊，穷尽分类成本极高。但"做不到完美MECE"不等于"放弃结构"。

初学者常犯这个错的根本原因是：**混淆了"罗列所有因素"和"结构化论证"**。

正确做法：
1. **先分层，再归类**：如果8个因素可以归为3个大类（如"外部竞争""用户质量""内部运营"），就用3个大类作为塔身第一层，每个大类下再展开2-3个具体因素。这变成子金字塔结构，而非扁平罗列。
2. **强制控制在5个以内**：认知心理学4±1组块。超过5个，读者无法同时处理。如果确实有8个独立因素，说明你的结论粒度太粗，应该拆分为2个子主题，各写一份报告。
3. **重叠的处理**：如果两个因素确实高度相关，合并为一个；如果相关性中等，明确标注"与因素X存在交互"，但不单独列为独立论据。

决策规则：
- **如果塔身超过5个论据** → 必须合并或拆分，不允许超过7个
- **如果论据之间存在重叠** → 先尝试归类为更高层级的抽象维度，用抽象维度作为塔身
- **如果无论如何都无法MECE** → 在方法说明中诚实声明分类边界，并说明"以下分类基于XX维度，存在一定程度的交叉"

### Q4: 探索性分析阶段，该不该强行套用金字塔原理？

**Hints:**
- 金字塔原理的局限性第一条是什么？
- 探索阶段的核心产出是什么——是"结论"还是"假设清单"？

**Answer:**
**不应该。** 金字塔原理要求结论先行，但探索阶段往往结论未定，强行套用会扼杀发现过程。这是[[stage-03-practice/07-document-writing/01-pyramid-principle/金字塔原理|金字塔原理]]明确标注的局限性。

探索阶段的正确结构：
- **问题清单 + 初步观察 + 待验证假设**
- 示例结构："当前发现3个异常方向：①搜索行为变化（具体数据）；②下单路径变化（具体数据）；③退款率变化（具体数据）。每个方向需要进一步验证的假设如下..."

决策规则：
- **如果数据已充分、结论已明确** → 用金字塔原理
- **如果处于探索阶段、结论未定** → 不用金字塔，改用"问题-假设-验证"结构
- **如果领导要求"必须有结论"但数据不充分** → 诚实写"当前数据无法支撑明确结论，建议补充XX分析后再做判断"——这也是一种结论，见[[stage-03-practice/07-document-writing/02-conclusion-writing/核心结论写作|核心结论写作]]中"结论要敢于说不知道"的经验

### Q5: 技术团队要"过程→结论"，业务方要"结论先行"，同一份报告怎么平衡？

**Hints:**
- 一份报告是否只能有一个版本？
- 思考"分层阅读"的设计——如何让不同读者各取所需

**Answer:**
**不要试图用一份报告满足两类读者。** 字节跳动内部实践是：同一分析结果产出三个版本——"技术版"（含代码和详细方法）、"业务版"（精简方法说明）、"高管版"（几乎无方法说明，聚焦结论和建议）。

如果只能产出一份报告，采用"分层结构"：
1. **首页**：纯金字塔结构，一页纸结论 + 3个支撑要点。业务方看完这一页即可决策。
2. **正文**：按技术团队的"实验设计→数据结果→显著性检验→结论"顺序展开，但每个章节开头用一句话总结该章节的核心发现。
3. **附录**：详细方法、代码、原始数据。

决策规则：
- **如果技术团队是主要评审方** → 正文用技术顺序，但首页强制放一页"执行摘要"（金字塔结构）
- **如果业务方是主要决策方** → 正文用金字塔结构，技术细节放附录
- **如果两方权重相当** → 产出两个版本，或采用"分层结构"让双方各取所需

关键原则：不要让读者为你的结构选择买单。技术团队可以跳过首页直接看正文，业务方可以只看首页不读正文——这才是好的文档设计。
